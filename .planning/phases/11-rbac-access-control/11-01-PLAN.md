1. Rewrite `firestore.rules` to membership-based access:
   - Keep `rules_version = '2'`
   - Helper `isAuth()`: `request.auth != null`
   - Helper `isProjectMember(projectData)`: returns true if `request.auth.uid == projectData.ownerId` OR `request.auth.uid in projectData.collaborators` (per D-11: collaborators is a map field, `uid in map` checks key existence)
   - Helper `getUserRole(projectData)`: returns `'productor'` if uid == ownerId, else `projectData.collaborators[request.auth.uid]`
   - Project documents (`match /projects/{projectId}`):
     - `allow read`: `isAuth() && isProjectMember(resource.data)`
     - `allow create`: `isAuth() && request.resource.data.ownerId == request.auth.uid` (creator becomes owner)
     - `allow update`: `isAuth() && isProjectMember(resource.data)` (any member can update; role-based write restrictions enforced in Cloud Functions and client)
     - `allow delete`: `isAuth() && resource.data.ownerId == request.auth.uid` (only owner per D-02)
   - Subcollections (`match /{subcollection}/{docId}` under projects):
     - `allow read, write`: `isAuth() && isProjectMember(get(/databases/$(database)/documents/projects/$(projectId)).data)` (uses 1 get() call)
   - Nested subcollections (`match /{subcollection}/{docId}/{nestedSub}/{nestedDocId}`):
     - Same rule as subcollections
   - Invitations collection (`match /invitations/{invitationId}`):
     - `allow read`: `isAuth() && resource.data.inviteeEmail == request.auth.token.email` (invitee can read their own invitations)
     - `allow write`: `if false` (only Cloud Functions via Admin SDK can create/update invitations)
   - Organizations rules: KEEP EXISTING (already correct from Phase 10)
   - User profiles: KEEP EXISTING (already correct from Phase 10)
   - Legacy ERPI settings: KEEP EXISTING read-only fallback
   - Security events collection (`match /security_events/{eventId}`):
     - `allow read, write`: `if false` (only Cloud Functions write; per D-17 audit trail)

2. Update `storage.rules`:
   - Keep default deny-all for unmatched paths
   - Project files (`match /projects/{projectId}/{allPaths=**}`):
     - `allow read, write`: `request.auth != null` (Storage rules cannot call Firestore get(), so membership enforcement remains at Cloud Functions layer per Phase 10 decision; auth-gate is the maximum enforcement possible)
   - Add comment explaining that Storage rules cannot check project membership (no Firestore get() in Storage rules) and that membership is enforced by Cloud Functions before upload

3. Upgrade `functions/src/auth/requireAuth.ts`:
   - Modify `requireProjectAccess` to check BOTH ownership AND collaborators map
   - Current: `if (data.ownerId !== uid) throw`
   - New logic:
     - If `data.ownerId === uid`, return `'productor'` as the role string
     - Else if `data.collaborators && uid in data.collaborators`, return `data.collaborators[uid]` as the role string
     - Else throw `HttpsError('permission-denied', 'No tienes acceso a este proyecto.')`
   - Change return type from `Promise<FirebaseFirestore.DocumentData>` to `Promise<{ role: string; projectData: FirebaseFirestore.DocumentData }>` so callers can access both the role and the project data
   - Update the JSDoc to reflect membership check instead of ownership check
   - Add a new helper `requireRole(role: string, allowedRoles: string[])` that throws `HttpsError('permission-denied', 'No tienes permiso para realizar esta accion.')` if the role is not in the allowed list. This is for future use when specific Cloud Functions need role-restricted access.
